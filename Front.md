# Об Архитектуре фронта

На фронте FSD - Feature-Sliced Design архитектура - организация кода по бизнес фичам, а не тех слоям.

# Почему именно оно?

1. У нас много бизнес фич
2. Параллельная разработка
3. Масштабируемость и отключение фич
4. Тестируемость -  фичи можно тестировать отдельно друг от друга без танцев с бубном
5. Сервисы в беке - фичи отлично синхронизируются с сервисами на беке

# App.svelte

Стартовая точка приложения, содержит в себе Router и общую структуру

## Router

Конфигурация маршрутов. Определение URL. В обычном Svelte 5 нет способов управления роунгом, поэтому юзаем доп либу -  **svelte-routing**.

# Guards

Определение уровня доступа юзера к сайту(admin/guest/auth)

# Layouts

Обертка для будущей страницы меняется в зависимости от уровня доступа(Guards)

# Pages

Непосредственно сами страницы на которые пользователя кидает маршрутизатор

# Features

 Здесь содержатся бизнес -  фичи
 Навигатор
 Интерактивный редактор для навигатора
 Расписание
 поиск преподов
 столовка
 Клиент жабы

Отдельно стоит сказать про **навигатор**. Для него будет использоваться **Three.js**, однако для лучшей интеграции со Svelte и упрощенной работы будет использоваться доп утилита **Threlte**.

## Threlte

фрелти - библиотека для свелта, которая упростит нам работу с навигатором, предоставит реактивность и интерактивность, доступ к событиям на уровне Svelte и полную поддержку TS.

# Shared

Переиспользуемые компоненты -  используются на нескольких страницах одновременно, позволяют не дублировать код, взаимодействовать с бекендом(API), хранить различные утилиты и библиотеки используемые глобально.

## Stores

Для управления состоянием объектов будут использоваться **руны** со Svelte 5 локально и **Svelte-stores** для глобального управления состоянием
*Почему нельзя все через руны?*
Потому что **Svelte stores** юзается глобально для данных юзера, темы, уведомлений.
А руны Svelte 5 для локального состояния внутри фичи.

## Немного про UI

Из доп утилит для UI - Tailwind CSS(по сути основа UI), даст максимальную гибкость, использование готовых классов, минимальный размер CSS документа(удаляется мусор при сборке). А также в качестве надстройки - Skeleton UI - библиотека готовых UI компонентов, которые будут кастомизироваться с помощью Tailwind CSS. При этом размер CSS документа также зависит от Tailwind CSS.

# API Gateway

API Gateway -  по сути прослойка между фронтендом и бекендом, а точнее между фронтом и микросервисами/единая точка входа фронта в бэк

## Какие проблемы решает?

1. роутинг, перенаправление запросов фронта на микросервисы
2. Аутентификация, а точнее проверка токена 1 раз для всех последующих маршрутов, то есть каждый сервис не будет проверять токен
3. Логирование
4. При смене адреса сервиса не нужно обновлять фронтенд, так как фронт не общается с беком напрямую
5. Rate limitting(ограничение кол-во запросов от 1 юзера)
